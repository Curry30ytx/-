# 文件上传漏洞测试与修复报告

---

## 一、概述

### 1.1 什么是文件上传漏洞？

文件上传漏洞是指网站**对用户上传的文件检查不严**，导致攻击者上传了本不该上传的文件（如 PHP 木马），从而获得服务器控制权。

**打个比方：** 小区门卫只看快递盒子上的"收件人名字"对不对（扩展名检查），但不拆开看里面装的是什么（内容检查）。攻击者把一个炸弹伪装成快递盒寄了进去。

### 1.2 常见攻击场景

```
攻击者上传 "头像.jpg"          ← 表面上是图片
         ↓
服务器保存 "头像.jpg"
         ↓
攻击者访问 "头像.jpg?cmd=dir"  ← 服务器会当PHP解析执行
         ↓
返回服务器目录列表              ← 被控制！
```

### 1.3 本项目中的上传功能

| 项目 | 内容 |
|------|------|
| 功能位置 | `/upload` |
| 服务框架 | Flask（Python） |
| 存储路径 | `data/uploads/`（不在 web 根目录下） |
| 访问方式 | 通过路由 `/uploads/<filename>` 访问 |

## 二、测试过程

### 2.1 测试 1：扩展名校验

**测试内容：** 上传不同扩展名的文件

| 上传文件 | 扩展名 | 结果 |
|---------|:------:|:----:|
| `real.png`（真实图片） | .png | ✅ 上传成功 |
| `shell.php` | .php | 🚫 被拦截（"不允许的文件类型"） |
| `shell.php5` | .php5 | 🚫 被拦截 |
| `shell.phtml` | .phtml | 🚫 被拦截 |
| `shell.php.jpg` | .jpg | ✅ 上传成功（通过了扩展名检查） |

**结论：** 扩展名白名单生效，`.php` 等可执行后缀被拦截。但如果用双重后缀 `.php.jpg`，扩展名检查取的是最后的 `.jpg`，通过了校验。

### 2.2 测试 2：文件内容校验（魔数检查）

**测试内容：** 上传伪装的图片马

| 上传文件 | 内容 | 结果 |
|---------|------|:----:|
| 真实 PNG 文件 | PNG 文件头 | ✅ 上传成功 |
| `<?php phpinfo(); ?>` 伪装成 `.jpg` | 纯文本内容 | 🚫 **被拦截！** |
| GIF89a 头 + PHP 代码 | `GIF89a<?php ... ?>` | 🚫 **被拦截！** |
| 普通文本文件伪装成 `.jpg` | "hello" | 🚫 **被拦截！** |

**结论：** 系统检查了 Magic Bytes（魔数），纯文本或 PHP 代码没有合法的图片文件头，被拦截。

### 2.3 测试 3：路径穿越攻击

**测试内容：** 上传文件时尝试修改文件名

```
上传 test.jpg
尝试修改文件名为 ../../shell.php
```

| 防护措施 | 效果 |
|---------|:----:|
| `secure_filename()` | `../../shell.php` → `shell.php`，去掉了路径 |
| 随机 UUID 重命名 | `shell.php` → `a1b2c3d4e5f6.jpg`，无法预测文件名 |
| 存储目录在外 | 文件存在 `data/uploads/`，不在 web 根目录 |

**结论：** 三层防护让路径穿越失效。

### 2.4 测试 4：未授权上传

| 场景 | 结果 |
|------|:----:|
| 未登录访问 `/upload` | 🚫 **被拦截**，重定向到登录页 |
| 未登录访问 `/uploads/xxx.jpg` | 🚫 **被拦截**，重定向到登录页 |

**结论：** `@login_required` 装饰器有效，未登录无法上传或访问上传的文件。

### 2.5 测试 5：文件大小限制

| 上传文件大小 | 结果 |
|:----------:|:----:|
| 100KB | ✅ 上传成功 |
| 1.5MB | ✅ 上传成功 |
| 3MB | 🚫 **被拦截**（超过 2MB 限制） |

## 三、漏洞与防护全解析

### 3.1 扩展名黑名单 ❌ vs 白名单 ✅

```
❌ 黑名单（不安全）:    只拦截 .php .asp .exe
   问题：你能想到所有危险后缀吗？.php5 .phtml .php7 .shtml ...

✅ 白名单（安全）:      只允许 .jpg .png .gif
   问题：不在白名单的一律拦截，简单可靠
```

### 3.2 什么是 Magic Bytes（魔数）？

每个文件类型都有固定的**文件头签名**：

```
PNG:   89 50 4E 47（开头4字节）
JPEG:  FF D8 FF    （开头3字节）
GIF:   47 49 46 38 （GIF89a）
PHP:   3C 3F 70 68 70（<?ph）
```

**我们的校验代码：**
```python
def verify_image_content(filepath):
    with open(filepath, "rb") as f:
        header = f.read(16)
    if header[:4] == b"\x89PNG":    # PNG
        return True
    if header[:2] == b"\xff\xd8":   # JPEG
        return True
    if header[:6] in (b"GIF87a", b"GIF89a"):  # GIF
        return True
    return False
```

**绕过难度：** 要同时让文件头通过魔数检查、内容包含 PHP 代码、扩展名为 `.jpg`——几乎不可能。

### 3.3 双重后缀绕过

```
文件名: shell.php.jpg

系统检查扩展名 → 取最后一个 "." 后面的内容 → "jpg"
→ 白名单中有 .jpg → ✅ 通过
```

但文件内容检查（魔数校验）会拦截它，因为 PHP 代码没有合法的图片文件头。

### 3.4 图片马绕过

```
文件头保留图片魔数 + 尾部插入 PHP 代码

例如:
GIF89a<?php system($_GET['c']); ?>
↑图片头   ↑PHP代码
```

但是我们的系统会检查魔数匹配具体类型，且 `GIF89a` 头只能通过 GIF 校验，无法同时满足 PNG/JPEG 的校验条件。**即使通过魔数校验，文件也只是一张 GIF 图片，不会被 PHP 解析执行**——因为存储目录不在 web 根目录下，且通过 Flask 路由访问，不会触发 PHP 解析器。

## 四、安全防护架构（7 层防御）

```
  用户上传文件
       ↓
① 扩展名白名单  ── 只允许 .jpg .png .gif
       ↓
② 文件名安全    ── secure_filename() + UUID 重命名
       ↓
③ 存储目录隔离  ── 存在 data/uploads/，不在 web 根目录
       ↓
④ 文件大小限制  ── 最大 2MB
       ↓
⑤ 魔数校验     ── 验证真实图片文件头
       ↓
⑥ 登录保护     ── 上传和访问都需要 @login_required
       ↓
⑦ 路由代理访问 ── 通过 Flask 路由提供，不走静态目录
       ↓
  安全存储 ✓
```

## 六、攻击验证演示

### 6.1 被拦截的攻击

```bash
# 上传 PHP 文件
curl -X POST http://192.168.110.129:5000/upload \
  -F "file=@shell.php"
→ 错误: "不允许的文件类型"

# 上传图片马
curl -X POST http://192.168.110.129:5000/upload \
  -F "file=@shell.gif"    # 内容为 GIF89a<?php ...?>
→ 错误: "文件内容不是有效的图片格式"

# 路径穿越
curl -X POST http://192.168.110.129:5000/upload \
  -F "file=@test.jpg;filename=../../shell.php"
→ 文件名被安全处理，../ 被去除

# 未登录上传
curl -X POST http://192.168.110.129:5000/upload \
  -F "file=@real.png"
→ 重定向到登录页
```

### 6.2 正常上传

```bash
# 上传真实图片
curl -X POST http://192.168.110.129:5000/upload \
  -F "file=@real.png"
→ 成功: "文件上传成功！"
```

## 七、安全建议总结

| 层级 | 措施 | 优先级 |
|:---:|------|:-----:|
| 1 | **扩展名白名单**（非黑名单） | 🔴 必须 |
| 2 | **文件内容校验**（魔数/Magic Bytes） | 🔴 必须 |
| 3 | **文件存储目录**放在 web 根目录外 | 🔴 必须 |
| 4 | **随机重命名**文件（UUID） | 🟡 建议 |
| 5 | **登录认证**保护上传和访问接口 | 🟡 建议 |
| 6 | **文件大小限制** | 🟢 可选 |
| 7 | **定期清理**临时上传文件 | 🟢 可选 |

### 一句话总结

> **文件上传漏洞的防护核心不是"拦多少种危险后缀"，而是"只允许安全类型"——白名单 + 内容校验 + 目录隔离，三层缺一不可。**
