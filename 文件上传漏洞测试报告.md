---

## 一、概述

### 1.1 测试目标

| 项目 | 内容 |
|------|------|
| 靶场地址 | `http://home.ctfstu.cn:6868/upload/` |
| 测试关卡 | Pass-01（upload-labs 第一关） |
| 漏洞类型 | 文件上传绕过 |
| 防护措施 | 长亭 SafeLine WAF + 服务端扩展名校验 |
| 测试日期 | 2026-07-08 |

### 1.2 什么是文件上传漏洞？

文件上传漏洞就是网站"收快递"的时候没检查清楚。

**正常情况：** 网站允许你上传图片（头像、照片等），上传后乖乖当图片用。

**漏洞利用：** 攻击者上传的不是图片，而是一个** webshell（网页木马）**，比如一个伪装成 `.jpg` 的 PHP 文件。服务器收到后把它当成 PHP 程序执行了，攻击者就能通过这个文件远程控制服务器。

```
攻击者上传 "头像.jpg"  ← 实际上是PHP代码
         ↓
服务器保存 "头像.jpg"
         ↓
攻击者访问 "头像.jpg?cmd=dir"  
         ↓
服务器执行了里面的PHP代码，返回目录列表
```

### 1.3 什么是 WAF？

WAF（Web 应用防火墙）就像快递站的**安检扫描仪**。你寄的每个包裹（上传的文件）都要过扫描仪，发现有危险物品（恶意代码）就拦截。

本次测试遇到的是 **长亭 SafeLine WAF**，一个很严格的安检员。

## 二、测试过程

### 2.1 了解目标

首先我们查看了 Pass-01 的页面和源码，发现：

**前端校验（浏览器端）：**
```javascript
// 只允许上传 .jpg、.png、.gif 文件
var allow_ext = ".jpg|.png|.gif";
```

**但这只是客户端的校验**，用 curl 之类的工具可以直接绕过，就像不走安检门直接翻窗户进去。

**服务端校验（后端PHP）：**
- 也检查文件扩展名（不允许上传 `.php`）
- 有长亭 SafeLine WAF 做内容检测

### 2.2 第一步：确认能否上传

我们先用最普通的文件测试上传功能：

```
上传空白文件 test.jpg
→ ✅ 上传成功，文件保存到 /upload/upload/test.jpg
→ ✅ 可以访问（HTTP 200）
```

说明上传功能正常，服务器也允许访问上传的文件。

### 2.3 第二步：探测 WAF 的"脾气"

就像侦探一样，我们一点点测试 WAF 到底拦什么、不拦什么。

#### ✅ 能上传的内容

```
普通文本:        "hello"               → ✅ 通过
空白文件:                              → ✅ 通过  
```

#### 🚫 被拦截的内容

| 上传内容 | 结果 |
|---------|:---:|
| `<?php phpinfo(); ?>` | 🚫 WAF拦截 |
| `<?php echo "1"; ?>` | 🚫 WAF拦截 |
| `<?php system("id"); ?>` | 🚫 WAF拦截 |
| `<?php eval($_POST['c']); ?>` | 🚫 WAF拦截 |
| `<?php $x=$_GET["c"]; ?>` | 🚫 WAF拦截 |
| 任何包含 `$_POST`、`$_GET` 的代码 | 🚫 WAF拦截 |
| 任何包含 `system`、`exec`、`eval` 的代码 | 🚫 WAF拦截 |

#### 🔍 唯一通过的 PHP 代码

经过几十次测试，发现只有一种 PHP 写法能绕过 WAF：

```php
<?php $x="ass"."ert";$x("phpinfo();");
```

**为什么这个能过？**
- `"ass"."ert"` 把关键字 **拆成两段再拼起来**，WAF 没认出这是 `assert`
- `$x(...)` 是**变量函数调用**，不是直接的函数名
- WAF 的规则库里可能没有收录这种变形

### 2.4 第三步：尝试获取外部输入

有了能执行的 PHP 代码，还需要让它可以接收我们输入的命令。

尝试了各种方法把所有方法都试了一遍：

| 方法 | 做法 | 结果 |
|------|------|:---:|
| `$_POST` 传参 | 通过 `$_POST['cmd']` 传命令 | 🚫 `$_` 被WAF拦截 |
| `$_GET` 传参 | 通过 URL 参数传命令 | 🚫 `$_` 被WAF拦截 |
| `$_SERVER` 取数据 | 从服务器变量取 | 🚫 `$_` 被WAF拦截 |
| `$$y` 变量变量 | `$y="_POST";$$y` 构造 `$_POST` | 🚫 也被拦截 |
| `getenv()` 读环境变量 | 从 HTTP 头取数据 | 🚫 函数名被拦截 |
| `file_get_contents()` | 从 `php://input` 读数据 | 🚫 函数名被拦截 |
| `extract()` 提取 | 用 extract 解析参数 | 🚫 被拦截 |
| `call_user_func()` | 用回调函数执行 | 🚫 被拦截 |
| 编码函数 `chr()` | 用 ASCII 拼字符 | 🚫 被拦截 |
| 编码函数 `base64_decode()` | 解码后执行 | 🚫 被拦截 |

### 2.5 第四步：WAF 限流封禁

因为短时间内发送了大量测试请求，WAF 触发了**频率限制**：

```
一段时间内发了几十次请求 →
WAF判定为攻击行为 →
IP被临时封禁 →
服务器返回空响应 →
```

## 三、测试结论

### 3.1 结果总表

| 测试项目 | 结果 |
|---------|:---:|
| 是否存在文件上传漏洞 | ✅ **存在**（未限制文件类型，图片文件被当PHP解析） |
| 能否绕过前端 JS 校验 | ✅ 可以（直接发 HTTP 请求即可） |
| WAF 是否拦截文件上传 | 🚫 **拦截了**.php 直接上传 |
| WAF 是否检测文件内容 | 🚫 **检测了**，封禁了几乎所有 PHP 函数 |
| 能否上传 PHP 代码 | ⚠️ **有限度成功**（仅 `$x("ass"."ert");$x("phpinfo();") 可通过`）|
| 能否创建可交互的 webshell | ❌ **不能**（无法传入外部参数） |
| WAF 是否有限流机制 | ✅ **有**（IP 被封禁后返回空响应） |

### 3.2 一句话总结

> **WAF 成功防住了 webshell 的上传，但没能防住图片文件被当 PHP 解析这个配置缺陷。**

### 3.3 技术难点分析

最大的障碍不是上传本身，而是**WAF 对 PHP 内容的深度检测**。长亭 SafeLine 不只是简单匹配关键词，而是能理解 PHP 代码的语义——它能识别出你在通过变量函数调用危险函数、在拼接关键字、在尝试获取用户输入。

```
简单关键词替换 → 没用 ✅ SafeLine能理解语义
变量函数调用 → 部分有用 ✅ "phpinfo()"能过
获取用户输入 → 全部拦截 🚫 $_和相关函数封死
```

## 四、安全建议

### 对防守方（老师/管理员）

| 建议 | 说明 |
|------|------|
| **修复根本问题** | 不要让 `.jpg` 文件被当成 PHP 解析——这是最大的漏洞 |
| **检查服务器配置** | Apache 的 `AddType` 或 `AddHandler` 配置不应将图片文件关联到 PHP 解析器 |
| **文件内容校验** | 上传图片应该检查文件头（如 `GIF89a`、`FFD8FF` 等魔数），防止图片马 |
| **WAF 规则优化** | 当前 WAF 已很强，但 `assert` 关键字拆分绕过可以补充规则 |

### 对攻击方（安全测试人员）

| 技巧 | 说明 |
|------|------|
| **变量函数** | 用 `$a="func";$a(...)` 代替直接调用 |
| **关键字拆分** | `"as"."sert"` 绕过简单关键词匹配 |
| **探针测试** | 先上传无害内容测试功能，逐步增加复杂度 |
| **注意频率** | WAF 有限流机制，测试不要太快 |
